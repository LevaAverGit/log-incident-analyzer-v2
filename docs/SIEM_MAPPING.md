# SIEM Mapping

This document describes how the tool's detection logic and output relate
to SIEM concepts, and how the JSON report can be integrated with a SIEM platform.

---

## Conceptual Mapping

The Log Incident Analyzer implements a simplified version of logic that runs
inside a SIEM's correlation engine:

| This tool | SIEM equivalent |
|---|---|
| Log parsers (`parser.py`) | Log source connectors / normalization |
| Detection rules (`detectors.py`) | Correlation rules / use cases |
| Finding with severity and score | Alert with severity level |
| Incident grouping by source IP | Entity-based correlation (IP as entity) |
| Multi-indicator bonus | Multi-event correlation rule (AND condition) |
| Timeline builder | Event sequence view / attack chain |
| JSON report | SIEM export / alert payload for SOAR |
| YAML thresholds (`config/`) | Rule tuning parameters |

## Mapping to SIEM Use Cases

Each detection rule in this tool corresponds to a standard SIEM use case
used in SOC environments:

| Tool rule | SIEM use case name | Common SIEM category |
|---|---|---|
| `ssh_brute_force` | SSH Brute Force | Authentication / Account Takeover |
| `web_scanning` | Web Directory Enumeration | Reconnaissance |
| `sensitive_path_access` | Sensitive File Access Attempt | Data Exposure / Recon |
| `suspicious_user_agent` | Malicious Tool Detected | Reconnaissance / Exploitation |
| `repeated_auth_errors` | Repeated Authentication Failures | Authentication |

## JSON Report Structure

The `--format json` output follows a flat structure that maps directly to
SIEM alert fields:

```json
{
  "generated_at": "2026-05-24T14:33:00",
  "summary": {
    "total_events": 229,
    "total_findings": 9,
    "total_incidents": 5,
    "critical_count": 2,
    "high_count": 1,
    "medium_count": 1,
    "low_count": 1
  },
  "incidents": [
    {
      "source_ip": "203.0.113.45",
      "severity": "critical",
      "score": 90,
      "finding_types": ["ssh_brute_force"],
      "findings": [
        {
          "finding_type": "ssh_brute_force",
          "severity": "critical",
          "score": 90,
          "evidence": {
            "attempt_count": 312,
            "usernames_targeted": ["root", "admin", "ubuntu"]
          },
          "recommendation": "Block source IP at firewall."
        }
      ]
    }
  ],
  "timeline": [...]
}
```

### Ingestion options

**ELK / Opensearch:** Use Filebeat or Logstash to push the JSON report file
as a document to an index. The `source_ip`, `severity`, and `finding_type`
fields are indexable and filterable.

**Splunk:** Use the `json` source type. Map `source_ip` to the `src` field
and `severity` to the `alert.severity` field using a field alias or transforms.conf.

**MaxPatrol SIEM / KUMA:** The JSON can be forwarded via a custom connector or
syslog relay. Format each incident as a structured event with the `source_ip`,
`severity`, and `finding_type` fields visible to the correlation engine.

## Detection Completeness vs. a Full SIEM

This tool covers a subset of what a production SIEM provides:

| Capability | This tool | SIEM |
|---|---|---|
| Log normalization | auth.log, nginx, syslog | Hundreds of connectors |
| Real-time streaming | No (batch file analysis) | Yes |
| Cross-source correlation | By source IP only | Full entity graph |
| Threat intelligence enrichment | No | IP reputation, IOC feeds |
| Long-term retention | No | Months to years |
| Alerting / notifications | No | Email, Slack, ticket integration |
| Compliance reporting | No | PCI-DSS, 152-FZ, GDPR templates |
| ML-based anomaly detection | No | Yes (in most platforms) |

This tool is a learning implementation of the detection layer only.
In production, the equivalent output is the alert payload generated by
a SIEM correlation rule, not the SIEM itself.

## Extending Detection to Sigma Rules

The detection rules in `detectors.py` can be expressed in [Sigma](https://github.com/SigmaHQ/sigma)
format for portability across SIEM backends. Example for `ssh_brute_force`:

```yaml
title: SSH Brute Force Attempt
status: stable
description: Detects repeated failed SSH login attempts from a single source IP
logsource:
  product: linux
  service: auth
detection:
  selection:
    message|contains: 'Failed password for'
  condition: selection | count(src_ip) > 10
falsepositives:
  - Misconfigured SSH clients
  - Internal scanners
level: high
```

Sigma rules can be compiled to Splunk SPL, KUMA, Elastic EQL, and other
backend formats using the `pySigma` library.
